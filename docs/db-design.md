# DB design

Single table, H2 in-memory by default. No joins, no FKs, no
migrations — this is a homework-scale CRUD service, not a production
schema.

## Schema

Table `books`:

| Column         | SQL type     | Nullable | Default | Source                | Notes                                    |
| -------------- | ------------ | -------- | ------- | --------------------- | ---------------------------------------- |
| `id`           | `BIGINT`     | no       | AUTO    | `@Id @GeneratedValue` | Primary key, identity-generated.         |
| `title`        | `VARCHAR(200)` | no     | —       | `@Column(nullable=false)` | Bean validation: `@NotBlank @Size(max=200)`. |
| `author`       | `VARCHAR(120)` | no     | —       | `@Column(nullable=false)` | Bean validation: `@NotBlank @Size(max=120)`. |
| `isbn`         | `VARCHAR(20)`  | yes    | —       | `@Column(unique=true)`   | Unique when present. May be null.       |
| `published_year` | `INT`       | yes      | —       | field `publishedYear` | Bean validation: `@Min(0)`.              |
| `created_at`   | `TIMESTAMP`  | no       | —       | `@PrePersist`         | Set on first save, never updated.       |

The Java field `publishedYear` is mapped to the column `published_year`
by Spring Boot's default `SpringPhysicalNamingStrategy`
(camelCase → snake_case). Same for `createdAt` → `created_at`.

## Indexes

- **Primary key** on `id` — engine default.
- **Unique** on `isbn` — implicit from `@Column(unique = true)`.

No secondary indexes. At homework scale a `findAll()` paged scan is
fast enough; if the row count grew, the obvious next index would be
on `title` / `author` for substring search (which is not a current
feature — see `docs/design.md` "Patterns considered and rejected").

## DDL (H2 reference)

```sql
CREATE TABLE books (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    title           VARCHAR(200) NOT NULL,
    author          VARCHAR(120) NOT NULL,
    isbn            VARCHAR(20),
    published_year  INT,
    created_at      TIMESTAMP NOT NULL,
    CONSTRAINT uk_books_isbn UNIQUE (isbn)
);
```

## Switching to PostgreSQL or MySQL

No code changes required — the entity uses standard JPA annotations
and Spring Boot's naming strategy. The only changes are in
`src/main/resources/application.properties` and `pom.xml`:

```properties
# PostgreSQL example
spring.datasource.url=jdbc:postgresql://localhost:5432/bookcatalog
spring.datasource.username=...
spring.datasource.password=...
spring.jpa.database-platform=org.hibernate.dialect.PostgreSQLDialect
```

```xml
<!-- pom.xml: replace the H2 runtime dependency -->
<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
    <scope>runtime</scope>
</dependency>
```

For MySQL, swap the driver artifact and dialect class. The
`BIGINT AUTO_INCREMENT` syntax in the reference DDL above is H2
flavoured; on PostgreSQL use `BIGSERIAL` or `BIGINT GENERATED BY
DEFAULT AS IDENTITY`, on MySQL use `BIGINT AUTO_INCREMENT`. Hibernate
emits the right DDL from the `@GeneratedValue(strategy = IDENTITY)`
mapping, so you do not need to write it by hand.

## What this schema does **not** model

- No `users` table — auth was intentionally out of scope (see
  `docs/spec.md` "Out of scope").
- No soft-delete column — deletes are hard, and the row goes away.
- No audit columns other than `created_at` (no `updated_at`,
  `created_by`, `updated_by`).
- No version column for optimistic locking — concurrent edits are
  last-write-wins, acceptable for a single-user homework CRUD.
